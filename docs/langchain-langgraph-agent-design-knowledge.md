# LangChain / LangGraph Agent 项目可迁移设计方法论

## 资产定位

这份知识资产面向后续 AI 复用，用来沉淀 LangChain 与 LangGraph 生态中 Agent 项目的通用设计方法。它不复述具体实现，也不作为项目说明书，而是抽取项目样本中反复出现的架构判断：什么时候用链，什么时候用图，什么时候引入工具、Skill、中间件、检查点、多智能体、RAG 与人工审核。

适用场景包括：研究型 Agent、数据分析 Agent、文档问答 Agent、多工具 ReACT Agent、多模态问答系统、可扩展 Skill Agent，以及需要长流程编排或多人机协作的 LangGraph 应用。

## 它解决的核心问题

LangChain / LangGraph Agent 项目通常不是单一“问答模型调用”问题，而是在不确定输入下组织模型、工具、状态和外部系统协作的问题。项目中的不同样本共同解决了以下几类痛点。

1. **多步骤任务不可控**：复杂任务需要规划、执行、汇总、校验，而不是一次模型生成。LangGraph 的节点和边把隐式推理流程外显为可追踪的状态机。
2. **工具数量增长导致选择困难**：当工具变多，模型容易误选、漏选或被工具描述淹没。Skill、Loader Tool、Tool Selector 和中间件把工具暴露从“全量开放”改为“按需激活”。
3. **上下文膨胀与长对话退化**：多轮对话、搜索结果和工具输出会持续增加 token 压力。摘要中间件、检查点、Store 与文件式状态把短期上下文、长期记忆和中间产物分层管理。
4. **外部知识接入不稳定**：RAG、MCP、搜索、数据库、PDF、多模态文件都属于外部上下文入口。项目把它们包装为工具或消息内容块，使模型通过统一协议使用外部能力。
5. **高风险动作缺少治理**：发消息、写文件、执行代码、调用外部 API 等动作需要审批、限制或回滚。Human-in-the-loop、interrupt、工具边界和检查点为风险节点提供暂停与恢复机制。
6. **多角色协作边界模糊**：研究、计算、写作、发布、审核等能力不应全部堆在一个 Agent Prompt 中。主管模式、分层模式、网络移交和协调器-工作器模式把角色职责变成可组合的协作拓扑。

## 总体设计思路

项目体现出一条从轻到重的设计路径：

1. **简单转换用 Chain**：输入到输出关系明确、步骤稳定、无需工具循环时，用 LCEL 链表达 prompt、model、parser 的线性组合。
2. **开放工具任务用 Agent**：用户目标明确但执行步骤不固定时，用 `create_agent` 或 ReACT Agent，让模型在工具调用循环中自行推进。
3. **业务流程用 Graph**：当流程有阶段、分支、循环、并行、人工审核或恢复要求时，用 LangGraph 显式定义状态、节点和边。
4. **复杂能力用 Skill / 中间件**：当工具体系持续扩张时，把能力注册、加载、过滤、说明、权限和状态更新交给 Skill 层与中间件，而不是继续扩大单个 Agent 的工具列表。
5. **跨角色协作用多智能体**：当任务天然需要不同职责、不同上下文或不同质量标准时，再引入 Supervisor、Swarm、Deep Agent 或协调器-工作器模式。

这条路径的关键原则是：**先保持单元简单，再用图和中间件治理复杂性，而不是一开始就把所有能力放进一个“大而全”的 Agent。**

## 可迁移的模块拆分方法

### 1. 模型接入层

职责是屏蔽模型供应商差异，给上层提供统一的 ChatModel 接口。项目中既有标准模型初始化，也有自定义模型封装和多供应商兼容接入。

可迁移方法：

- 把模型初始化从业务节点中剥离出来，避免每个节点硬编码模型、密钥和服务地址。
- 为不同用途保留模型选择空间，例如通用对话、推理、结构化输出、多模态理解、摘要压缩可以使用不同模型策略。
- 中间件可以根据运行上下文切换模型，但模型切换逻辑应被视为运行时策略，而不是节点业务逻辑。

### 2. Prompt 与 Chain 层

职责是处理明确的语言转换任务，例如分类、提取、摘要、改写、计划生成、报告生成。项目中的 Prompt、Parser 和结构化输出共同构成“模型输出契约”。

可迁移方法：

- 对需要被后续节点消费的输出，优先使用结构化输出，而不是让后续节点解析自然语言。
- Chain 适合做局部确定性处理，例如“生成计划”“提取字段”“把检索结果压缩为摘要”。
- Prompt 不应承担流程控制；流程控制应交给 Graph 的边、条件函数或 Command。

### 3. 状态层

职责是定义 Agent 协作期间哪些信息会被保留、合并、覆盖、传递和恢复。项目中同时使用了消息状态、TypedDict/Pydantic 状态、Reducer、检查点和 Store。

可迁移方法：

- 简单聊天可以只维护消息列表；一旦流程涉及计划、检索结果、工具产物、人工审核、草稿、引用或任务状态，就应定义显式状态结构。
- 状态字段应表达“协作契约”，例如输入、阶段性产物、决策结果、外部证据、最终输出，而不是塞入所有临时变量。
- 并行或多工作器场景需要设计合并规则，避免多个节点写同一状态时互相覆盖。
- 需要恢复的流程必须配合检查点，并用会话标识区分不同用户或任务线程。

### 4. 工具层

职责是把外部能力变成模型可调用的动作。项目中的工具覆盖搜索、数据库、文件、代码执行、绘图、PDF 解析、MCP 服务、天气查询等不同类型。

可迁移方法：

- 工具应该是窄接口，描述清楚“何时调用、输入是什么、返回什么、不能做什么”。
- 工具只负责能力边界内的动作，不负责整条业务流程。流程组合应由 Agent 或 Graph 完成。
- 对高风险工具要增加审批、权限、沙箱、路径限制、结果校验或只读模式。
- 工具返回应偏结构化、可读、可被后续节点继续消费；避免把过大的原始内容直接塞回消息历史。

### 5. Skill 与能力注册层

职责是把一组相关工具、说明和元数据封装为可发现、可激活、可过滤的能力包。项目中的 Skill 设计体现了“先让 Agent 看到加载器，再按需暴露实际工具”的思路。

可迁移方法：

- 每个 Skill 应包含元数据、Loader Tool、实际工具集合和使用说明。
- Loader Tool 始终可见，实际工具按需可见，降低工具选择复杂度。
- Registry 负责发现和管理 Skill；Middleware 负责在模型调用前根据状态过滤工具。
- Skill 状态可以采用替换、累积或 FIFO 等策略。长期会话适合累积，工具很多的场景适合限制同时激活数量。

### 6. 中间件层

职责是运行时治理，而不是业务处理。项目中的中间件用于工具选择、摘要压缩、待办规划、人工审批、文件系统能力和动态模型/工具切换。

可迁移方法：

- 中间件适合处理横切关注点：工具可见性、上下文压缩、审批、运行时策略、文件/记忆后端、观测日志。
- 不要把领域业务逻辑塞进中间件，否则会削弱图结构的可读性。
- 中间件应尽量通过运行时状态和请求覆盖机制工作，保持 Agent 主流程简洁。

### 7. 图编排层

职责是把任务拆成节点，把节点之间的依赖、分支、循环、并行和终止条件显式化。项目中的 LangGraph 样本覆盖了线性图、条件图、循环图、路由、并行、评估优化、协调器-工作器和人工审核。

可迁移方法：

- 节点表示职责单元，边表示决策逻辑。不要让一个节点同时承担分类、检索、生成、审核和发送。
- 条件边适合表达“根据状态选择下一步”，Command 适合在节点内同时更新状态并跳转。
- 并行 fan-out/fan-in 适合相互独立的子任务；需要为合并节点设计清晰的聚合规则。
- 循环必须有可解释的停止条件，必要时配合递归限制或计数状态，避免无限工具/节点循环。

### 8. RAG 与外部上下文层

职责是把非模型内知识转为可引用、可追踪、可压缩的上下文。项目中既有 PDF 文本切分、向量库检索、Retriever Tool，也有多模态消息块和引用提取。

可迁移方法：

- 文档处理、切分、索引、检索、引用生成应拆成独立模块，而不是混入对话节点。
- 检索结果需要保留来源元数据，最终回答应能回溯到文档块或外部来源。
- 对扫描件、图片、音频等非文本输入，应先转换为模型可接受的内容块或中间文本，再进入问答链路。
- RAG 工具不应只返回“答案”，更应返回支撑答案的上下文证据。

### 9. 多智能体协作层

职责是在多个专业角色之间分配任务、移交上下文和合成结果。项目中包含主管、分层主管、网络移交、协调器-工作器和 Deep Research 子代理等模式。

可迁移方法：

- 多智能体不是默认选项，只有当职责天然分化、上下文隔离有价值或并行能显著提升效率时才引入。
- Supervisor 适合中心化调度；Swarm/网络模式适合角色之间动态移交；协调器-工作器适合从计划动态生成多个同构子任务；分层模式适合大型任务按团队边界拆分。
- 子 Agent 的输入应窄而清晰，避免把完整全局上下文无差别传给每个角色。
- 最终合成应由明确的合成节点或主管完成，避免多个 Agent 直接争夺最终输出权。

### 10. API / UI 边界层

职责是把 Agent 能力以服务形式暴露，同时处理流式输出、文件上传、多模态内容、历史消息转换和错误返回。

可迁移方法：

- API 层只负责协议转换和传输，不应承载核心推理流程。
- 流式输出需要区分增量内容、完成事件、错误事件和附加元数据。
- 多模态输入要统一为消息内容块，避免每个节点单独处理文件协议。
- UI 需要的引用、图片路径、生成物标识等，应在服务层做格式化，但证据来源仍由 RAG/工具层提供。

## 关键协作关系

### Chain 与 Graph 的关系

Chain 是节点内部的局部能力，Graph 是节点之间的全局编排。一个稳定的设计是：每个图节点内部可以使用一个或多个 Chain，但 Chain 不直接决定全流程跳转。

这样做的好处是：局部语言任务可复用，整体流程可观测、可恢复、可测试。

### Agent 与 Tool 的关系

Agent 负责决定“是否需要调用工具、调用哪个工具、如何解释工具结果”；Tool 负责执行边界清晰的外部动作。工具描述越明确，Agent 的行为越稳定。

如果工具数量较少，可以直接绑定到 Agent；如果工具数量增长，应引入工具选择中间件或 Skill 加载机制。

### State 与 Message 的关系

Message 适合保存对话轨迹，State 适合保存流程产物。复杂 Agent 不应只依赖消息历史传递所有信息，否则节点之间会产生隐式耦合。

一个好的状态设计会把“用户输入、模型决策、中间证据、待处理任务、最终产物”分层保存，让每个节点只读取自己需要的状态。

### Graph 与 Human Review 的关系

人工审核不是额外的 if/else，而是图中的正式节点。需要审核时，流程中断并保存状态；恢复时，人的决策作为 Command 回到图中继续执行。

这种设计适合邮件发送、文件写入、外部系统变更、支付、发布、数据库写操作等高风险节点。

### Skill 与 Middleware 的关系

Skill 定义能力包，Middleware 控制能力包何时可见。Registry 解决“有哪些能力”，State 记录“当前激活了哪些能力”，Middleware 在每次模型调用前根据状态裁剪工具列表。

这种协作关系可以把工具规模从“模型一次看到全部工具”降为“模型只看到当前任务相关工具”。

### RAG 与 Agent 的关系

RAG 不只是问答前置步骤，也可以作为 Agent 的工具。前者适合固定知识库问答，后者适合让模型在需要时主动检索、比较、引用和继续追问。

如果任务要求可溯源，RAG 层必须保留文档块、来源和引用编号，生成层只负责使用这些证据组织答案。

## 设计模式提炼

### 1. 线性 Chain 模式

解决问题：输入输出关系稳定，流程不需要分支和工具循环。

为什么这样设计：LCEL 把 Prompt、Model、Parser 组合成可读的流水线，适合小而确定的语言转换任务。

迁移经验：把线性 Chain 放在图节点内部，作为局部处理器，而不是让 Chain 承担全局流程。

### 2. 结构化输出决策模式

解决问题：自然语言决策难以被程序稳定消费。

为什么这样设计：分类、计划、评估、路由结果都需要机器可读格式，结构化输出把模型判断变成显式契约。

迁移经验：凡是会影响下游分支、工具选择或状态更新的模型输出，都应优先结构化。

### 3. ReACT 工具循环模式

解决问题：用户目标明确，但完成路径依赖工具反馈。

为什么这样设计：模型可以在观察工具结果后继续推理，适合搜索、查询、计算、文件读取等开放执行任务。

迁移经验：给 ReACT Agent 设置工具边界、递归限制和错误处理，不要让工具循环无限扩张。

### 4. 显式 StateGraph 流程模式

解决问题：业务流程有固定阶段、状态产物和可恢复要求。

为什么这样设计：Graph 把隐式 prompt 流程变成显式节点，使流程更容易审计、扩展和调试。

迁移经验：一旦任务有分支、循环、并行、人工审核或阶段产物，就优先考虑 LangGraph。

### 5. 路由模式

解决问题：不同输入类型需要进入不同处理链路。

为什么这样设计：先用轻量分类器决定路径，再交给专门节点处理，避免所有节点都理解所有任务。

迁移经验：路由节点应输出有限枚举；每个路由目标应有明确职责和终止条件。

### 6. 提示链加门控模式

解决问题：生成结果需要逐步增强，但并非每次都要走完整流程。

为什么这样设计：先生成，再用门控条件判断是否需要改写、润色、补充或结束。

迁移经验：门控逻辑应尽量可解释；如果门控依赖模型判断，使用结构化输出承载判断结果。

### 7. 评估器-优化器模式

解决问题：一次生成质量不稳定，需要反馈驱动迭代。

为什么这样设计：生成器负责产出，评估器负责给出是否接受和修改方向，循环直到达标或触发停止条件。

迁移经验：适合写作、代码生成、数据分析解释、报告润色等质量型任务；必须限制迭代次数。

### 8. 并行 fan-out / fan-in 模式

解决问题：多个子任务互相独立，可以并行完成后聚合。

为什么这样设计：并行节点减少等待时间，聚合节点统一整理输出风格和结构。

迁移经验：并行前要确认子任务无共享写冲突；并行后要有合并规则和冲突处理策略。

### 9. 协调器-工作器模式

解决问题：子任务数量运行时才知道，且每个子任务结构相似。

为什么这样设计：协调器先生成任务清单，再动态分发给同构工作器，最后合成。

迁移经验：适合章节写作、批量分析、批量检索、批量评估；工作器输入要小，输出要可合并。

### 10. Supervisor 模式

解决问题：多个专业 Agent 需要中心化调度。

为什么这样设计：主管负责判断顺序和移交，专业 Agent 只处理自己的能力范围。

迁移经验：适合“先研究再计算再写作”这类有自然顺序的任务；主管 prompt 应明确每个 Agent 的职责边界和完成条件。

### 11. 分层 Supervisor 模式

解决问题：大型任务中角色太多，单层主管难以管理。

为什么这样设计：先按团队划分，再由顶层主管协调团队，降低单个主管的决策复杂度。

迁移经验：适合企业流程、复杂报告、端到端业务自动化；团队边界应按产物或能力划分，而不是按文件数量划分。

### 12. 网络移交模式

解决问题：对话中当前负责角色可能动态变化。

为什么这样设计：每个 Agent 既能处理自己的职责，也能通过 handoff 工具把任务交给更合适的角色。

迁移经验：适合多角色助手、客服分流、专家会诊；需要检查点保存当前激活角色和对话上下文。

### 13. Deep Research 模式

解决问题：研究型任务需要计划、检索、反思、分工、引用和综合报告。

为什么这样设计：研究不是简单搜索，而是有限预算下的证据收集与综合。计划器定义研究方向，子研究员隔离上下文，反思工具控制继续搜索还是停止，写作节点统一引用和结构。

迁移经验：适合市场研究、技术调研、竞品分析、资料综述；关键是设置搜索预算、停止条件和引用合并规则。

### 14. 按需 Skill 模式

解决问题：工具生态扩展后，Agent 工具面过大。

为什么这样设计：Agent 先调用加载器理解某类能力，再看到该能力下的具体工具，使工具暴露从静态全量变为动态局部。

迁移经验：适合插件化 Agent、企业内部工具平台、领域能力市场；每个 Skill 要有清晰说明、权限声明和工具集合。

### 15. Human-in-the-loop 模式

解决问题：某些自动化动作需要人类确认。

为什么这样设计：interrupt 把人工决策纳入图状态，而不是在流程外临时处理。

迁移经验：适合发送、发布、写入、执行、采购、删除、权限变更等高风险动作；需要检查点保证审核前后状态连续。

## 架构取舍逻辑

| 任务形态 | 优先方案 | 不宜优先使用 | 取舍逻辑 |
| --- | --- | --- | --- |
| 单步问答或格式转换 | Chain | 多智能体 | 简单任务不要引入编排成本 |
| 需要结构化提取或分类 | Chain + structured output | 自然语言解析 | 让下游消费稳定契约 |
| 需要工具但流程开放 | ReACT / create_agent | 固定图 | 工具反馈决定下一步时让 Agent 循环处理 |
| 有明确业务阶段 | StateGraph | 单 Agent prompt | 显式流程比隐式提示更可控 |
| 有分支、循环、人工审核 | StateGraph + Command / interrupt | 线性 Chain | 流程控制应成为图的一部分 |
| 工具数量多且可分组 | Skill + Middleware | 全量工具绑定 | 动态工具可见性降低误调用 |
| 子任务可并行 | 并行图或协调器-工作器 | 单线程 Agent | 独立子任务适合 fan-out/fan-in |
| 多职责专家协作 | Supervisor / 分层 / Swarm | 单一大 Prompt | 角色职责需要显式边界 |
| 文档或外部知识问答 | RAG Tool / Retriever | 纯模型回答 | 用可追溯证据补足模型知识 |
| 高风险外部动作 | Human-in-the-loop | 全自动工具调用 | 人类审批是架构节点，不是异常分支 |

## 可迁移设计原则

### 1. 先设计状态，再设计节点

状态决定协作边界。先回答“哪些信息需要跨节点存在、哪些信息需要恢复、哪些信息要合并”，再设计节点会更稳。否则节点容易通过消息文本隐式传递信息，导致流程难测试、难恢复。

### 2. 节点只做一种职责

一个节点可以调用模型、工具或 Chain，但它应该只有一个清晰目的，例如分类、检索、生成、评估、聚合、审核、发送。职责越单一，条件边和状态更新越容易理解。

### 3. 用结构化输出承载决策

凡是会影响程序流向的模型输出，都不应只靠自然语言。路由、计划、评分、审核建议、任务清单等应使用结构化结果，必要时做校验和 fallback。

### 4. 工具是能力边界，不是流程容器

工具应该执行一个外部动作或查询，不应把多轮业务流程塞进工具内部。流程属于 Graph，能力属于 Tool。

### 5. 工具可见性需要治理

少量工具可以直接绑定；工具变多后要引入 Tool Selector、Skill Loader 或 Middleware。模型看到的工具越少、越相关，调用质量越高。

### 6. 把长期记忆、短期上下文和工作文件分开

短期对话适合消息历史；可恢复流程适合检查点；跨会话偏好适合 Store；大中间产物适合文件或外部存储。不要把所有内容都塞进 prompt。

### 7. 并行前先确认独立性

并行适合互不依赖的检索、章节生成、候选方案生成或批量评估。不适合共享可变状态、需要严格顺序或会产生写冲突的任务。

### 8. 循环必须有停止条件

ReACT 循环、评估优化循环、搜索循环都需要预算、最大次数或收敛条件。停止条件应写进图或中间件策略，而不是只写在提示词里。

### 9. 人工审核应成为一等节点

对于高风险动作，应在图中设计审核节点、中断点和恢复路径。这样既能保留自动化效率，又能保留责任边界。

### 10. 证据和引用从一开始设计

只要项目涉及 RAG、搜索或研究报告，就应在文档块、检索结果和最终回答之间保留引用关系。后补引用通常不可靠。

### 11. 多智能体按职责拆，不按热闹程度拆

引入多智能体的理由应是职责边界、上下文隔离、并行收益或质量治理，而不是“看起来更智能”。能用一个图节点解决的，不必拆成多个 Agent。

### 12. Prompt 负责语义，Graph 负责控制

Prompt 可以告诉模型如何理解任务，但不应承担系统级控制流。阶段、分支、循环、审批、恢复和终止应由 Graph 显式表达。

## 迁移到其他项目时的设计流程

1. **定义任务拓扑**：判断任务是线性转换、工具循环、固定流程、并行任务、多角色协作，还是高风险自动化。
2. **定义状态契约**：列出跨步骤保存的信息、需要合并的信息、需要恢复的信息、最终产物和证据来源。
3. **拆分节点职责**：把模型生成、检索、工具调用、评估、聚合、审核、外部动作拆成最小可理解节点。
4. **选择编排方式**：简单用 Chain，开放工具用 Agent，确定业务流程用 StateGraph，多角色协作用 Supervisor/Swarm/Deep Agent。
5. **设计工具边界**：为每个外部能力定义输入、输出、错误、权限和安全限制；避免工具承担流程。
6. **治理工具规模**：当工具超过模型容易稳定选择的范围时，引入 Skill、Tool Selector 或中间件过滤。
7. **设计上下文策略**：决定哪些信息留在消息历史，哪些进入状态，哪些进入 Store，哪些写入文件或外部数据库。
8. **设计停止与恢复**：为循环设置预算，为长流程设置检查点，为高风险动作设置 interrupt。
9. **设计证据链**：如果答案依赖外部知识，确保检索结果、来源元数据和最终引用关系贯通。
10. **设计观测与验证**：输出中间状态、工具调用、路由决策、评估反馈和最终证据，便于调试与质量评估。

## 不建议直接迁移的做法

项目中有些写法适合教学演示，但迁移到新的 Agent 项目时需要抽象、替换或加固。

1. **不要把配置写进节点或工具**：模型供应商、密钥、外部服务地址、数据库连接、文件根目录都应进入配置层或环境层，业务节点只依赖抽象后的客户端。
2. **不要让代码执行工具裸奔**：Python 执行、Shell 执行、文件写入、数据库访问都需要沙箱、权限、超时、路径限制和审计；生产场景应默认最小权限。
3. **不要依赖全局变量传递业务数据**：演示中为了方便会把数据放在全局环境，新项目应使用状态、Store、临时工作区或显式数据句柄。
4. **不要让模型一次看到所有工具**：工具数量增加后，应使用 Skill、Tool Selector 或中间件裁剪工具面，避免模型误选和上下文浪费。
5. **不要只用消息历史承载流程状态**：复杂流程需要显式状态字段，否则后续节点会从自然语言里反向推断中间结果，稳定性差。
6. **不要把 API/UI 逻辑与 Agent 编排混在一起**：接口层只做协议转换、流式传输、文件接入和错误包装；核心推理流程应可被 CLI、服务端和测试共同复用。
7. **不要缺省信任外部检索结果**：搜索、RAG、MCP、数据库和文件解析都要保留来源、错误状态和可信度边界，最终回答要能说明依据来自哪里。
8. **不要把循环控制只写在 Prompt 里**：工具循环、评估循环和研究循环都应有程序侧预算、递归限制或停止条件。

## 给后续用 LangChain / LangGraph 生态写 Agent 项目的设计指导原则

1. **从流程复杂度选择技术栈**：能用 Chain 就不要上 Graph；需要分支、循环、并行、恢复或人工审核时再上 LangGraph；需要开放工具探索时使用 Agent；需要职责隔离时再多智能体化。
2. **每个 Agent 项目都先画状态图**：在写 prompt 前，先明确输入、状态、节点、边、终止条件和失败路径。状态图是 Agent 架构的骨架。
3. **模型决策必须可消费**：分类、路由、计划、评分、任务拆分等结果应结构化，避免后续节点从自然语言里猜测意图。
4. **模块按职责而不是技术名拆分**：推荐拆为模型适配、状态定义、工具/Skill、图节点、编排图、记忆/存储、RAG/外部上下文、API/UI、评估与安全治理。
5. **把工具当作受控外设**：工具需要最小权限、清晰描述、错误返回和安全边界。写文件、执行代码、访问数据库、调用外部服务等工具默认需要限制或审批。
6. **工具多时采用分层暴露**：基础加载器常驻，领域工具按需激活；中间件按状态裁剪工具；必要时用工具选择器限制模型每轮可见工具数。
7. **Graph 节点要小，边要显式**：节点负责产物，边负责方向。不要让 prompt 暗含流程分支，也不要让一个节点隐式完成多个阶段。
8. **并行与多智能体只用于真实独立性**：如果子任务共享上下文、共享写状态或需要严格顺序，就不要并行。并行收益来自独立任务，不来自盲目拆分。
9. **长任务必须有预算意识**：搜索次数、工具调用次数、评估循环次数、上下文大小和输出长度都要可控。复杂 Agent 的稳定性来自边界，而不是无限自主。
10. **人类介入点要提前设计**：凡是不可逆、外部可见、涉及权限或可能产生损失的动作，都应在图中设计审核、拒绝、修改和恢复路径。
11. **RAG 先设计引用，再设计回答**：文档块、来源、页码、检索结果和回答引用要从数据结构上贯通。不要在最终回答阶段临时拼接来源。
12. **把中间产物视为资产**：计划、检索摘要、评估反馈、草稿、引用表和工具结果都应能被后续节点复用，也应能用于调试和复盘。
13. **Prompt 保持职责清晰**：系统提示定义角色、边界和输出要求；流程顺序交给图；工具选择交给 Agent 和中间件；安全策略交给工具封装和审核节点。
14. **优先可恢复，再追求自动化**：检查点、线程隔离、Store 和 interrupt 是生产级 Agent 的基础能力。没有恢复机制的长流程自动化很难可靠落地。
15. **从小闭环演进到大系统**：先做单节点或单 Agent 的可验证闭环，再加入状态、工具、RAG、检查点、多智能体和中间件。每增加一层复杂性，都应对应一个明确问题。
